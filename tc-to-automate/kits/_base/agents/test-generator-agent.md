# Test Generator Agent

**Single Responsibility**: Transform `locators.json` + parsed test cases + kit configuration
into complete, compilable test automation artifacts using kit-specific templates.

---

## Inputs (passed by calling command)

| Parameter | Type | Description |
|---|---|---|
| `mode` | string | `"pom-only"` \| `"spec-only"` \| `"full"` \| `"bdd-full"` |
| `locatorsFile` | string | Path to `locators.json` |
| `testCases` | object[] | Parsed test cases (from test-case-parser skill) |
| `kitConfig` | object | Full contents of `kit.config.json` |
| `bdd` | boolean | Whether to generate BDD artifacts |
| `outputDir` | string | Base output directory |
| `projectName` | string | Project subdirectory name |
| `overwrite` | boolean | Whether to overwrite existing files |

---

## Pre-Generation Steps

1. Read `kitConfig.templates` path → verify template directory exists
2. Read `locatorsFile` → parse component map
3. Read `tc-to-automate/kits/_base/skills/test-case-parser.md` → parse `testCases` input into structured objects
4. Read `tc-to-automate/kits/_base/skills/template-renderer.md` → understand template syntax
5. Read `tc-to-automate/kits/<kitId>/KIT.md` → apply kit-specific naming conventions

---

## Mode: `pom-only`

Generate ONE Page Object file per component in `locators.json`.

For each `ComponentName` in locators.json:
1. Load template: `<kitConfig.templates>/page-object.ts.tmpl` (or `.java.tmpl` for Java kits)
2. Build template context:
   ```
   { className: "ComponentName", locators: [...], imports: [...] }
   ```
3. Render template → write to `<outputDir>/<projectName>/pages/<ComponentName>.ts`
4. Skip if file exists AND `overwrite = false`

**Validation after pom-only mode**: Run `tsc --noEmit` (TS) or `mvn compile` (Java).

---

## Mode: `spec-only`

Generate ONE spec file per test case. Assumes POMs already exist.

For each test case in `testCases`:
1. Identify which components/pages are involved
2. Load template: `<kitConfig.templates>/spec-nonbdd.ts.tmpl`
3. Build template context:
   ```
   { testId: "TC-001", title: "...", steps: [...], pages: ["LoginPage"], imports: [...] }
   ```
4. Render → write to `<outputDir>/<projectName>/tests/<testId>-<slugified-title>.spec.ts`
5. Verify: all `import` paths resolve to existing POM files; if not, abort and report missing POMs

---

## Mode: `full`

Run `pom-only` first, then `spec-only`. Both phases must succeed.

---

## Mode: `bdd-full`

Run `pom-only`, then for each test case generate a feature + step definition pair.

### Feature generation
1. Load template: `<kitConfig.templates>/spec-bdd.feature.tmpl`
2. Build context:
   ```
   { featureName: "Login", scenarios: [{ id, title, given, when, then, tags }] }
   ```
3. Render → write to `<outputDir>/<projectName>/features/<featureName>.feature`
4. Apply rules from `tc-to-automate/rules/bdd-gherkin.md`

### Step definition generation
1. Load template: `<kitConfig.templates>/step-definitions.ts.tmpl`
2. Build context: extract unique step text from all scenarios in this feature
3. Check `<outputDir>/<projectName>/steps/` for EXISTING step definitions — reuse any matching steps
4. Render only NEW steps → write to `<outputDir>/<projectName>/steps/<featureName>.steps.ts`
5. Import matching POM class for each step that requires UI interaction

---

## Template Rendering Rules

1. Apply template renderer from `tc-to-automate/kits/_base/skills/template-renderer.md`
2. All `{{variable}}` placeholders must be resolved — no undefined values in output
3. If a template variable cannot be resolved, log a warning and use a clearly marked TODO:
   `/* TODO: replace with actual value */`
4. Always add file header comment:
   ```typescript
   // Generated by automation-kit | Kit: kit-u1 | Do not edit manually
   // Regenerate with: /generate-automation-tests
   ```

---

## Post-Generation Validation

After generating all files in any mode:
1. Run `npx tsc --noEmit` (TypeScript) — all generated TS must compile with 0 errors
2. Confirm: no raw selector strings (e.g., `'[data-testid=...]'`) appear in spec files
3. Confirm: all POM imports in spec files resolve to actual generated files
4. If BDD: run `npx cucumber-js --dry-run` — all steps must be matched

Report results. Fix any import path errors automatically before reporting completion.

---

## Output Directory Structure

```
<outputDir>/<projectName>/
├── pages/
│   └── <ComponentName>.ts          ← one per component
├── tests/
│   └── <TC-ID>-<title>.spec.ts     ← one per test case (non-BDD)
├── features/
│   └── <feature>.feature            ← one per feature (BDD)
└── steps/
    └── <feature>.steps.ts           ← one per feature (BDD)
```
